## Research Findings: OpenClaw Delivery-Mirror + Ollama Timeout

### 1. What is "delivery-mirror"?

The term "delivery-mirror" is **not a formally documented OpenClaw component** in public docs. However, from the architecture research, OpenClaw uses a **Gateway** as its central hub that routes messages between channels (Telegram, TUI, etc.) and agent sessions. The message flow is:

**Telegram webhook --&gt; Gateway (message routing + delivery) --&gt; Agent session --&gt; LLM provider (Ollama)**

The "delivery-mirror" appears to be the Gateway's internal message delivery subsystem that handles streaming responses back to channels. It has a **hardcoded 60-second timeout** in the announce/delivery queue (`src/agents/subagent-announce.ts` - `sendAnnounce()` calls `callGateway()` with `timeoutMs: 60000`).

### 2. Root Cause of Your Timeout

There are **three compounding issues** in your config:

**Issue A -- Hardcoded HTTP timeout (~60s) in the LLM client layer.** This is separate from `agents.defaults.llm.idleTimeoutSeconds: 120`. The HTTP fetch that actually calls Ollama has a hardcoded ~60s timeout that ignores your config setting. This is a confirmed bug: [Issue #61487](https://github.com/openclaw/openclaw/issues/61487).

**Issue B -- Context injection bloat.** The Gateway injects system prompts, memory, tool definitions, and conversation history before sending to Ollama. [Issue #7725](https://github.com/openclaw/openclaw/issues/7725) shows that even for a short message like "Hey", the Gateway can inject 12k+ tokens of context. Your Gemma4 model at 56 tok/s would need significant time to process this, pushing past the hardcoded timeout.

**Issue C -- Timeout misclassified as rate_limit.** This is exactly your `candidate=ollama/gemma4:e4b reason=rate_limit` error. Confirmed bug: [Issue #5240](https://github.com/openclaw/openclaw/issues/5240) and [Issue #13336](https://github.com/openclaw/openclaw/issues/13336). When the LLM request times out, OpenClaw puts the provider into cooldown as if it were a cloud API rate limit, which makes no sense for local models.

**Why `--local` mode works:** In local/TUI mode, the agent runs directly without the Gateway relay layer, so the extra Gateway timeout and context injection overhead are bypassed.

### 3. Your Config Problems

In `E:/02_openclawAgent/workspace/config/openclaw.json`:

- **`api: "openai-completions"` with `/v1` baseUrl** -- This forces Ollama to use the OpenAI-compat shim. Streaming + tool calling is broken in this mode. The native `api: "ollama"` with baseUrl `http://host.docker.internal:11434` (no `/v1`) is recommended.
- **No `requestTimeout`** on the Ollama provider -- The provider-level timeout override is missing.
- **Cooldown config is insufficient** -- You have `billingBackoffHoursByProvider.ollama: 0.001` but this only reduces cooldown duration; it doesn't prevent the misclassification.

### 4. Recommended Fixes

**Fix 1 -- Switch to native Ollama API and add requestTimeout:**
```json
"ollama": {
  "baseUrl": "http://host.docker.internal:11434",
  "apiKey": "ollama",
  "requestTimeout": 180000,
  "models": [
    {"id": "gemma4:e4b", "name": "Gemma4 e4b 8B", "api": "ollama"},
    {"id": "gemma4:e2b", "name": "Gemma4 e2b 5B", "api": "ollama"}
  ]
}
```
Key changes: `baseUrl` drops `/v1`, `api` changes to `"ollama"`, `requestTimeout: 180000` (180s).

**Fix 2 -- Pre-load model with keep_alive to avoid cold-start:**
```bash
curl http://localhost:11434/api/generate -d '{"model":"gemma4:e4b","prompt":"hi","stream":false,"keep_alive":"60m","options":{"num_predict":1}}'
```

**Fix 3 -- Set OLLAMA_KEEP_ALIVE environment variable:**
```
OLLAMA_KEEP_ALIVE=-1
```
This prevents the model from being unloaded from VRAM, eliminating cold-start delays.

**Fix 4 -- If tool calling is not needed, disable streaming:**
Add `"params": {"streaming": false}` to each model config. This avoids the streaming protocol bug with Ollama.

**Fix 5 -- Clear stale cooldown state:**
Delete the `usageStats` section from `~/.openclaw/agents/main/agent/auth-profiles.json` (or equivalent path inside your Docker container), then restart the gateway.

### 5. api: "ollama" vs api: "openai-completions"

| Aspect | `api: "ollama"` | `api: "openai-completions"` |
|--------|-----|-----|
| Endpoint | `/api/chat` (native) | `/v1/chat/completions` (shim) |
| Streaming + Tool calling | Works simultaneously | Broken -- must disable one |
| baseUrl | `http://host:11434` | `http://host:11434/v1` |
| Timeout behavior | Same hardcoded issue but more reliable streaming | Streaming bugs compound timeout |
| Recommendation | **Use this** | Avoid unless proxy requires OpenAI format |

### 6. Known Open Issues (Unresolved)

The hardcoded ~60s HTTP timeout in the LLM client layer ([#61487](https://github.com/openclaw/openclaw/issues/61487), filed very recently) and the rate_limit misclassification ([#5240](https://github.com/openclaw/openclaw/issues/5240), [#13336](https://github.com/openclaw/openclaw/issues/13336)) are still open bugs. The `requestTimeout` provider-level config ([#43946](https://github.com/openclaw/openclaw/issues/43946)) was proposed as a fix but may not be fully implemented in all code paths yet.
