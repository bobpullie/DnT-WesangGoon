## 주요 교훈 (TEMS 등록 대상)

### 1. Plan 내부 모순 검출 책임
큰 설계 플랜에서 reference code ↔ plan test가 충돌할 수 있음. Plan writer의 mental dry-run 한계. 구현 세션에서 reference code를 plan test에 맞춰 수정하는 판단이 필요할 수 있음. (Task 1.4 I3/I4 `k<=1` 케이스)

### 2. Forward-looking code review의 가치
Task 1.1 code reviewer가 Phase 1.6에서 발현할 body `\n` 누적 버그를 사전 예측 → Option B 수동 패치 → Phase 1.6 idempotency 테스트에서 실제 검증됨. "5개 phase 후 fire할 버그"를 잡아낸 review의 전략적 가치 확인.

### 3. Windows 한글 locale 대응 관례
`Path.write_text("...한글...")` 은 cp949로 인코딩되어 UTF-8 reader가 fail. 모든 raw file I/O에 `encoding="utf-8"` 명시 필수. (Task 1.1~1.8 전반 관례)

### 4. Subagent 커밋 권한 통제 (TGL #77 적용)
Implementer subagent는 `git add` 까지만, `git commit` 은 controller. 첫 Task 1.1 fix에서 1 dispatch 낭비 후 이후 모든 프롬프트에 명시. "DO NOT run git commit. Stage and report."

### 5. `pythonpath = [".."]` load-bearing 결정
Plat 레이아웃(`atlas/scripts/`) + `from atlas.scripts.topology import` 임포트 요구 → `pythonpath = [".."]` + Python 3.3+ namespace package 조합으로 해결. `pyproject.toml`에 load-bearing 주석으로 보호 (나중에 누가 cleanup으로 지우지 않도록).

### 6. 2-stage review 간소화 판단
Plan이 매우 prescriptive (TDD + byte-exact reference)인 경우 formal 2-stage review는 중복. Task 0.1/1.1만 full review, 이후 controller git diff 검증만으로 충분. 단, 첫 태스크와 중요 primitive는 review 필수.
