# /meta-check — 의지/방법 메타 분석

## Usage
```
/meta-check <종일군의 요청 또는 아이디어>
```

## Purpose
종일군의 요청을 받으면 즉시 실행하기 전에, 요청의 **의지**(무엇을 이루고 싶은가)와 **방법**(어떻게 이루려 하는가)을 분리하여 분석합니다.
방법이 의지를 달성하기 위한 최적 수단인지 검증하고, 더 나은 대안이 있으면 제시합니다.

이 스킬은 TCL #22의 절차적 구현입니다.

## Execution Steps

### Step 1. 의지(Intent) 추출
종일군의 요청에서 **근본적으로 이루고 싶은 것**을 1~2문장으로 추출합니다.
- "~하고 싶다", "~해야 한다", "~가 필요하다" 형태
- 기술적 수단을 제거하고 순수한 목적만 남깁니다

### Step 2. 방법(Method) 추출
요청에 포함된 **구체적 실행 방법**을 추출합니다.
- 기술 선택, 구현 방식, 도구 지정 등
- 암묵적인 가정도 명시적으로 드러냅니다

### Step 3. 부합성 검증
방법이 의지를 위한 **최적/합리적/효율적** 수단인지 검증합니다:

| 검증 항목 | 질문 |
|-----------|------|
| 필요충분성 | 이 방법이 의지를 달성하기에 충분한가? 과하지 않은가? |
| 효율성 | 같은 의지를 달성하는 더 간단한 방법이 있는가? |
| 부작용 | 이 방법이 의도치 않은 부정적 효과를 만드는가? |
| 전제 조건 | 이 방법이 성립하려면 어떤 가정이 필요한가? 그 가정은 유효한가? |

### Step 4. 대안 제시 (해당 시)
부합성 검증에서 문제가 발견되면:
1. **문제점**을 구체적으로 설명
2. **대안**을 1~2개 제시 (각각의 장단점 포함)
3. **권장안**을 명시하되, 최종 결정은 종일군에게

### Step 5. 결과 보고
```
## Meta-Check Result
- **의지:** {추출된 의지}
- **방법:** {추출된 방법}
- **부합성:** ✅ 최적 | ⚠️ 개선 가능 | ❌ 재검토 필요
- **근거:** {검증 결과 요약}
- **대안:** {있으면 제시}
- **권장:** {최종 권장안}
```

## Hard Rules
- Anti-sycophancy (TCL #31): 종일군의 방법이 비효율적이면 솔직히 지적
- 대안 제시 시 반드시 **장단점 모두** 포함 (편향 방지)
- 의지가 불명확하면 실행하지 않고 종일군에게 확인 질문
