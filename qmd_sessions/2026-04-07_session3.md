### 2. Task 1 — 플랜 결함 발견

- 근본 원인: 플랜 전체가 `.scripts/` (선행 점) Python 패키지 사용

- Python: `import .scripts` 자체가 relative import 토큰이라 불가
- 종일군도 동일 증상으로 고생 중이었음 — spec 단계부터 누적된 결함

### 3. 근본 해결

- 디스크 디렉토리 rename + `[tool.setuptools] packages = ["khub"]` 추가
- git: `c7785c3 fix: rename .scripts package to khub`

## 교훈 (TEMS 후보)

### TGL 후보
**Python 패키지 디렉토리는 점으로 시작 금지.** PEP 517/setuptools 검증이 거부하고 import도 불가능하다. 시각적 숨김은 `[tool.setuptools] packages = ["name"]` 명시 + Obsidian `userIgnoreFilters` 같은 외부 메커니즘으로 처리.
- 트리거: pyproject 작성 시, 새 패키지 디렉토리 명명 시
- 근거: KnowledgeHub spec/plan 전체가 `.scripts/` 사용했다가 Task 1에서 첫 pip install 시 발견. 만약 spec 단계에서 잡혔으면 4300줄 플랜 작성 후 sed 일괄 치환할 필요 없었음

### TCL 후보
**대형 plan 실행 전 패키징 dry-run.** 26개 태스크짜리 plan을 작성했다면 Task 1 (project init)을 작성하는 즉시 사용자가 실제 `pip install -e .` 또는 `python -c "import package"`를 한 번 돌려서 골격이 import 가능한지 확인. 골격이 깨졌으면 후속 태스크 컨텍스트를 일제히 수정해야 함.
- 트리거: brainstorming/spec/plan 단계에서 패키징 결정이 들어올 때
- 근거: 이번 세션에서 종일군이 같은 결함으로 고생했음을 발견. spec 작성자(이전 세션 위상군)가 검증을 안 했음
