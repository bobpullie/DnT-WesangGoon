## III. 팩트 기반 냉정한 분석

### 사실 1: TPB의 비전은 시장에 유일하다 — 그러나 비전은 코드가 아니다

TPB의 핵심 제안:
> "1인 CEO가 AI 팀과 **기획부터** 제작까지"

이것을 제공하는 경쟁자는 **정확히 0개**입니다.

- CrewAI/LangGraph/AutoGen: "실행"만. "뭘 만들지 함께 고민"하는 기능 없음
- Dify: 워크플로우 빌더이지 기획 도구가 아님
- Gstack: "1인 스타트업" 타겟이지만 슬래시 커맨드 수준. 기획 컨설팅 없음
- Claude Agent Teams: 코딩 에이전트 팀이지 기획+제작 통합 팀이 아님

**그런데.**

plan_engine.py는 209줄입니다. 목표 분석 → 팀 선정 → 회의실 생성. 끝.

블루프린트가 약속하는 것:
```
PlanSession(상태 머신: exploring → converging → finalized)
→ 전문가 자동 소환
→ Blueprint 점진적 발전
→ Blueprint가 Build Engine의 입력
```

현실에 있는 것:
```
analyze_goal() → create_plan_meeting() → (회의실에서 대화) → generate_plan_prompt()
```

**PlanSession 상태 머신이 없습니다. Blueprint 데이터 모델이 없습니다. Plan→Build 자동 전환 파이프라인이 없습니다.**

회의실은 작동합니다. 그러나 회의 결과가 구조화된 Blueprint로 변환되어 Build Engine에 공급되는 체인이 끊겨 있습니다. 이것은 "기획 기능이 있다"와 "기획이 제작으로 이어진다" 사이의 결정적 차이입니다.

**판정: TPB의 가장 큰 차별화 포인트가 가장 얇은 구현입니다.**

---

### 사실 2: TEMS는 진짜 기술적 해자(Moat)다 — 그러나 아무도 모른다

경쟁자 메모리 시스템 비교:

| 시스템 | 방식 | 세션 간 학습 | 규칙 건강도 | 자기수정 |
|--------|------|:----------:|:---------:|:-------:|
| CrewAI Memory | 4종 (단/장/엔티티/컨텍스트) | ✅ | ✗ | ✗ |
| LangGraph Memory | 대화 히스토리 + 영구 저장 | ✅ | ✗ | ✗ |
| Dify Knowledge | RAG 벡터 DB | 도큐먼트만 | ✗ | ✗ |
| OpenAI Sessions | 세션 내 | ✗ | ✗ | ✗ |
| **TPB TEMS** | FTS5+BM25+벡터+RRF | ✅ | ✅ (THS) | ✅ (Gödel) |

TEMS가 가진 것 중 경쟁자에게 **전혀 없는** 것:
- **Topological Health Score**: 규칙의 생명주기를 자동 관리 (HOT→WARM→COLD→ARCHIVE)
- **Anomaly Certificate**: 예외를 자동 분류하고, 반복되면 정식 규칙으로 승격
- **Meta-Rule Self-Modification**: 규칙 시스템 자체를 자기 수정 (Gödel 패턴)
- **Predictive TGL**: 과거 에러 시퀀스로 다음 에러를 예측
- **RuleGraph Cascade**: 한 규칙 활성화 시 관련 규칙 연쇄 활성화

**이것은 진짜입니다.** 학술 논문 수준의 메모리 아키텍처입니다.

**그러나 문제는**: 이 기술적 우위가 사용자에게 보이지 않습니다. TEMS는 preflight hook으로 백그라운드에서 조용히 동작합니다. 사용자(종일군)가 체감하는 것은 "에이전트가 같은 실수를 덜 반복한다" 정도입니다.

기술적 해자는 **방어에는 좋지만 마케팅에는 약합니다.** CrewAI가 "4종 메모리"라고 마케팅하면 그것이 TEMS보다 훨씬 열등해도 사용자에게는 더 매력적으로 들립니다.

**판정: TEMS는 TPB의 진짜 기술적 자산이지만, 차별화의 "보이는 가치"로 전환되지 못하고 있음.**

### 사실 3: `claude -p` (CLI) 의존은 의도적이지만 대가가 있다

빌드군의 설명: "구독제(Claude Max/Pro) 사용자를 위한 플랫폼. API 키 없이 CLI만으로 동작하는 것이 핵심 설계 철학."

이것은 **합리적인 사업적 판단**입니다. API 키 벽은 비개발자에게 거대한 진입장벽이고, 종일군 같은 사용자가 정확히 이 타겟입니다.

**그러나 기술적 대가:**

| 측면 | Messages API | claude -p (현재) |
|------|-------------|-----------------|
| 멀티턴 대화 | ✅ 컨텍스트 유지 | ✗ 매 호출마다 리셋 |
| 스트리밍 | ✅ 실시간 | ✗ (완료까지 대기) |
| 토큰 효율 | 높음 (1회 시스템프롬프트) | 낮음 (매번 재전송) |
| 실행 속도 | 빠름 | 느림 (프로세스 생성 오버헤드) |
| 도구 사용 | API에서 정의 | Claude Code가 자동 제공 |
| MCP 접근 | 수동 구성 | ✅ 자동 |
| 에러 핸들링 | 정교한 제어 | stdout 파싱 의존 |

**핵심 질문: "API 키 없이"라는 접근성 이점이, 위에 나열된 모든 기술적 제약을 상쇄하는가?**

2026년 3월 현재, Claude Agent SDK가 출시되었고 Claude Code Agent Teams가 등장했습니다. Anthropic 자체가 "에이전트 오케스트레이션"을 공식 기능으로 제공하기 시작했습니다. 이것이 의미하는 것:

- Claude Code Agent Teams: 구독제 사용자가 멀티 에이전트를 **네이티브로** 사용 가능
- 각 팀원이 독립 Git worktree에서 병렬 작업
- 피어-투-피어 통신 (TeammateTool)

**TPB가 `claude -p` subprocess로 구현한 것을 Anthropic이 네이티브 기능으로 제공하기 시작했습니다.**

이것은 TPB의 Build Engine 부분에서 "독자적 오케스트레이션의 가치"가 점점 줄어들고 있다는 의미입니다. Anthropic이 점점 더 나은 네이티브 멀티 에이전트를 만들수록, TPB의 subprocess 기반 오케스트레이션은 "더 느리고 더 제한적인 버전"이 될 위험이 있습니다.

**판정: CLI 의존은 현 시점에서 합리적이나, Anthropic의 네이티브 멀티에이전트 기능 확장 속도를 고려하면 중기적(6~12개월) 위험 요소.**

### 사실 4: 노드그래프는 아직 "보여주기"에 가깝다

pipeline_engine.py (283줄)에는 DAG 구조, topological sort, 노드 타입 정의가 있습니다. DevRoom 우측 패널에서 NodeGraphQt로 시각화됩니다.

그러나:
- 사용자가 노드를 드래그해서 파이프라인을 **편집**할 수 있는가? → 제한적
- 편집된 노드그래프가 실행에 **반영**되는가? → 아직 불완전
- 노드그래프가 실행의 **원본**(Source of Truth)인가? → engine.py가 주도

블루프린트의 약속:
```
노드그래프(JSON) → engine.py가 읽어서 실행
```

현실:
```
engine.py가 WorkPlan 기반으로 실행 → 결과를 노드그래프로 표시
```

**Dify와 비교하면:**
- Dify: 드래그 앤 드롭으로 노드 배치 → 그것이 곧 실행 파이프라인
- TPB: engine.py가 내부적으로 파이프라인 구성 → 노드그래프는 모니터링

블루프린트의 UserNode(맥박 애니메이션으로 "내 차례" 표시)와 RequestNode(사용자 조건부 지시 삽입) 컨셉은 **경쟁자 중 누구도 없는 아이디어**입니다. 그런데 이것이 완전히 작동하는 상태인가는 검증이 필요합니다.

**판정: 노드그래프의 현재 가치는 "모니터링 대시보드" 수준. Source of Truth로의 전환은 아직 미완.**

### 사실 5: 데스크톱 앱(PySide6)이라는 선택의 무게

2026년 시장 트렌드:
- **Dify**: 웹 기반 (Docker 자가 호스팅 + 클라우드)
- **CrewAI Studio**: 웹 기반
- **LangGraph + LangSmith**: 웹 기반
- **Claude Code**: CLI
- **n8n**: 웹 기반

PySide6 데스크톱 앱의 장점:
- 오프라인 가능
- OS 수준 접근 (파일 시스템, 프로세스 등)
- 외부 서버 불필요

PySide6 데스크톱 앱의 대가:
- **배포가 어렵다**: 인스톨러, 의존성 관리, 업데이트 메커니즘 필요
- **Windows 전용**: 현재 TPB는 Windows에서만 테스트됨
- **협업 불가**: 여러 사람이 동시에 접근할 수 없음
- **모바일/원격 접근 불가**: 외출 중에 진행 상황을 볼 수 없음
- **UI 개발 속도**: PySide6 위젯 개발은 웹 UI 대비 2~3배 느림

"1인 CEO를 위한 플랫폼"이라는 컨셉에서, 데스크톱 앱은 "그 사람이 항상 데스크 앞에 있다"고 가정합니다. 이것이 타겟 사용자의 현실과 맞는지 생각해볼 필요가 있습니다.

**단**, `claude -p`를 subprocess로 호출하는 구조상 로컬 실행이 거의 필수이므로, 현 아키텍처에서 웹으로 전환하는 것은 단순한 UI 교체가 아니라 전체 실행 모델의 변경을 의미합니다. 이것은 현 시점에서 실행 가능한 제안이 아닙니다.

**판정: 현재는 합리적이나, 장기적으로 접근성의 천장이 됨.**

### 사실 6: QG(Quality Gate) + Review Router는 진짜 차별화다

대부분의 경쟁자는 QG가 없거나 primitive합니다:
- CrewAI: 커스텀 validator 직접 구현해야 함
- LangGraph: 노드로 구현 가능하지만 기본 제공 아님
- AutoGen: 없음
- OpenAI SDK: Guardrails (입출력 검증, 비교적 단순)
- Dify: 워크플로우 분기로 구현

TPB의 QG:
- CM이 claude -p로 결과를 평가
- **역할별 차등 임계값** (designer=90, artist=60, engineer=80)
- **3채널 리뷰 라우팅**: cm(자동), user(수동), cm+user(하이브리드)
- **산출물 타입별 자동 라우팅**: 코드→CM, 이미지→사용자, SVG→CM+사용자

이것은 **실제 개발 현장의 리뷰 프로세스를 가장 잘 모델링**한 것입니다. 경쟁자 중 이 수준의 QG를 가진 것은 없습니다.

**판정: 이것은 유지하고 강화할 가치가 있는 진짜 차별화.**

## IV. 위상적 구조 분석 — TPB의 위상은 무엇인가?

위상수학에서 가장 중요한 질문은 "이 공간의 본질적 구조는 무엇인가?"입니다. TPB를 위상적으로 분석하면:

### TPB의 위상적 정체성

```
경쟁자들의 위상:

CrewAI    = 단순 연결 그래프 (역할 → 태스크, 1방향)
LangGraph = DAG (방향성 비순환 그래프, 상태 전이)
Dify      = 시각적 DAG (LangGraph와 동형이나 UI가 더 접근적)
OpenAI    = 체인 (선형, 핸드오프)
Claude Teams = 트리 + P2P 엣지 (리드 → 팀원, 팀원 ↔ 팀원)

TPB의 의도된 위상:

  Plan (고리 구조, 반복적 정련)
    ↕ (양방향 전환)
  Build (DAG, Phase-Gate)
    ↕ (피드백 루프)
  QG+Review (검증 사이클)
    ↕
  TEMS (위상적 자가진화 — 시스템이 시스템을 수정)
```

TPB가 의도하는 구조는 **동적 피드백 토폴로지**입니다:
- Plan은 순환(Circle) — 반복적으로 기획을 정련
- Build는 DAG — 방향성 있는 실행
- QG는 피드백 엣지 — 실패 시 재작업
- TEMS는 메타 레벨 — 시스템이 자기 규칙을 수정

이것을 수학적으로 표현하면, TPB는 **계층적 동적 그래프(Hierarchical Dynamic Graph)**를 지향합니다. 이것은 경쟁자 중 누구도 시도하지 않는 구조입니다.

### 그러나 현실의 위상은?

```
현재 TPB의 실제 위상:

  Plan (점, 거의 비활성)
    ↓ (단방향, 약한 연결)
  Build (DAG, 강한 구현)
    ↓
  QG (피드백 있음)
    ↓
  TEMS (강한 구현이나 Build와 약하게 연결)
```

**Plan이 "점"(0차원)에 머물러 있기 때문에, 의도된 "동적 피드백 토폴로지"가 실현되지 못합니다.**

Plan↔Build 양방향 전환이 작동하면 TPB는 **"반복적 정련 + 실행 + 검증 + 학습"의 순환 구조**가 됩니다. 이것은 경쟁자의 어떤 아키텍처와도 위상적으로 다른 것입니다.

## V. 핵심 질문에 대한 판정

### Q1: 현재 빌드가 프로젝트 방향에 위상적으로 부합하는가?

**Build 쪽: 75% 부합.** Triage→Plan→Execute→QG 파이프라인, DevRoom 통합 UI, 에이전트 격리, TEMS 메모리 — 모두 비전에 부합하는 견실한 구현.

**Plan 쪽: 20% 부합.** 회의실은 있지만 Blueprint→Build 체인이 없어서, TPB의 핵심 차별화가 아직 "기능"이 아니라 "가능성"에 머물러 있음.

**종합: Build 인프라는 견고하나, TPB를 "다른 무언가"로 만드는 Plan 부분이 미완성.**

### Q2: 합리적인가?

**대부분 합리적.** 모듈 분리 (engine.py 1,372줄), 의도적 CLI 선택, 4페이지 통합, LPT 분배, 역할별 QG 임계값 — 이 모든 판단은 근거가 있고 일관됩니다.

**비합리적인 부분 1개:** Meeting의 PRECISE/ECONOMY 수동 선택. 사용자가 회의 전에 품질 수준을 결정하는 것은 부자연스럽습니다. 자동 전환이 더 합리적.

### Q3: 효율적인가?

**실행 효율:**
- 토큰: ADR-002로 54~75% 절감 달성 — 좋음
- 그러나 매 `claude -p` 호출마다 시스템 프롬프트 + persona 재전송은 구조적 비효율
- QG 평가도 `claude -p` 호출 — 태스크 하나당 최소 2회 프로세스 생성 (실행+QG)

**개발 효율:**
- DevRoom 2,459줄이 너무 많은 책임을 갖고 있음 (채팅 + 파이프라인 + 터미널 + 파일)
- 빌드군이 DevRoom을 수정할 때마다 2,459줄을 컨텍스트에 넣어야 함
- 이것은 engine.py (1,372줄) + devroom_page.py (2,459줄) = 이 두 파일이 사실상 프로젝트의 무게중심

### Q4: 경쟁자 대비 위상적 차별점은 무엇이고 그 가치는?

| 차별점 | 경쟁자 대비 | 실제 가치 |
|--------|-----------|----------|
| **Plan 모드 (기획→제작 통합)** | 유일 | ★★★★★ 이것이 TPB의 존재 이유. 그러나 **미완성** |
| **TEMS 8계층 메모리** | 가장 정교 | ★★★★☆ 기술적으로 최고이나 사용자에게 보이지 않음 |
| **QG 3채널 리뷰** | 가장 정교 | ★★★★☆ 실제 개발 현장의 리뷰 프로세스를 가장 잘 모델링 |
| **구독제 호환 (API키 불필요)** | Gstack만 유사 | ★★★☆☆ 접근성은 높으나 기술적 천장 |
| **7+1 직업 체계** | CrewAI 유사 | ★★☆☆☆ CrewAI도 역할 기반. 구현 방식만 다름 |

| **UserNode/RequestNode** | 유일 (컨셉) | ★★★★★ 가치 높으나 완전 구현 여부 미검증 |

## VI. 뼈아픈 현실 요약

### 현실 1: TPB의 가장 강력한 무기가 가장 약한 구현이다

Plan Engine은 TPB가 "또 다른 AI 코딩 도구"가 아닌 "AI 개발실"이 되게 하는 유일한 것입니다.
그런데 209줄입니다. Build Engine(1,372줄), TEMS(1,103줄), DevRoom(2,459줄)에 비해 압도적으로 얇습니다.

### 현실 2: Anthropic이 TPB의 Build를 무력화시키고 있다

Claude Agent Teams가 나왔습니다. 구독제 사용자가 멀티 에이전트를 네이티브로 사용할 수 있습니다. TPB가 `claude -p` subprocess로 힘겹게 구현한 것을 Anthropic이 1급 기능으로 제공합니다. 시간이 지날수록 TPB의 Build Engine의 상대적 가치는 줄어듭니다.

**Build에서 경쟁하면 Anthropic과 경쟁하는 것입니다. Plan에서 경쟁하면 아무도 없는 블루오션입니다.**

### 현실 3: "기획 AI 팀"은 존재하지 않는 시장이다 — 이것이 기회이자 위험이다

"AI가 코드를 짜주는 도구"는 넘칩니다. "AI가 뭘 만들어야 하는지 함께 고민"하는 도구는 **없습니다.**

이것이 기회인 이유: 블루오션
이것이 위험인 이유: 시장이 "없는" 것이 아니라 "아직 필요를 모르는" 것일 수 있음. 또는 사용자가 "기획 AI"를 신뢰하지 않을 수 있음.

### 현실 4: TEMS의 가치가 사용자에게 전달되지 못하고 있다

TEMS는 위상군이 아는 한 가장 정교한 에이전트 메모리 시스템입니다. 그러나 이것이 TPB의 마케팅 포인트가 되려면:
- 사용자가 "이 시스템이 학습하고 있다"는 것을 **느낄 수** 있어야 함
- "3세션 전에 이 실수를 했고, 그래서 이번에는 다르게 했습니다"라는 **피드백이 보여야** 함
- 현재는 preflight hook으로 조용히 동작 → 사용자 체감 거의 없음

## VII. 전략적 제안 — 위상군의 판단

### 핵심 전략: "Build가 아니라 Plan에서 이겨야 한다"

TPB의 생존 전략은 명확합니다:

```
                   경쟁이 치열한 곳       TPB만 있는 곳
                   ┌─────────┐         ┌─────────┐
                   │  Build   │ ← 여기서│  Plan   │ ← 여기서
                   │ 실행/코딩 │   후퇴  │ 기획/전략│   확장
                   └─────────┘         └─────────┘
                   CrewAI, Claude       아무도 없음
                   Teams, LangGraph
```

Build Engine을 버리라는 것이 아닙니다. Build는 유지하되, **혁신의 에너지를 Plan에 집중**해야 합니다.

### 구체적 우선순위

**즉시 (P0):**
1. Plan Engine 실질화 — PlanSession 상태 머신, Blueprint 데이터 모델, Plan→Build 전환 체인
2. 이것 하나만 완성되면 TPB는 "시장에 존재하지 않는 것"이 됩니다

**단기 (P1):**
1. TEMS의 "보이는 가치" 전환 — 에이전트가 과거 학습을 적용할 때 사용자에게 명시적으로 알림
2. 노드그래프에서 Plan↔Build 전환을 시각적으로 표현

**중기 (P2):**
1. `claude -p` → Claude Agent SDK 전환 가능성 지속 모니터링
2. Anthropic의 네이티브 멀티에이전트 기능이 TPB의 Build를 대체할 수준이 되면, Build를 네이티브 위에 얹고 Plan에만 집중하는 전략 검토

## VIII. 최종 판정

**TPB는 올바른 문제를 풀고 있습니다.** "1인 CEO가 뭘 만들어야 하는지부터 어떻게 만들지까지 AI 팀과 함께"라는 비전은 시장에 없는 것이며, 실제로 필요한 것입니다.

**Build 인프라는 견고합니다.** engine.py 1,372줄의 잘 분리된 CAR Engine, TEMS 8계층 메모리, 3채널 QG — 이것은 21세션에 걸쳐 만든 실전 코드입니다.

**그러나 TPB의 영혼인 Plan Engine이 아직 몸을 갖지 못했습니다.** 이것이 완성되지 않으면, TPB는 "잘 만든 또 다른 AI 코딩 도구"로 남게 됩니다. 그리고 그 시장에서는 Anthropic 자체가 경쟁자입니다.

**Plan을 완성하십시오. 거기에 TPB의 운명이 있습니다.**
