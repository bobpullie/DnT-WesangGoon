### 2.2 TimeHighlight(Prediction Window) 시간축 고정 — 근본 수정

**3단계 진화를 거쳐 근본 원인 해결:**

1. **1차 시도 (submitKlineIndex):** `klineCount - 1`을 gameStore에 저장 → 실패. 원인: `MAX_KLINES=120` 슬라이딩 윈도우 + `setData()` 매번 호출 → logical index가 0부터 재설정되어 인덱스 매번 밀림
2. **2차 시도 (submitOpenTime):** 캔들의 `openTime`(timestamp, ms)을 gameStore에 저장 → 성공. `(submitOpenTime - firstKlineOpenTime) / 60000`으로 현재 배열 내 정확한 인덱스 역산
3. **좌표계 수정:** `containerWidth`(가격축 포함) → `timeScale().width()`(series pane만) → 줌 시 캔들과 window 위치 벌어짐 해결

**변경 파일:**
- `gameStore.ts` — `submitOpenTime` 필드 + `markPredictionSubmitted(lastKlineOpenTime)`
- `GameScreen.tsx` — `submitOpenTime`, `firstKlineOpenTime` props 전달
- `TimeHighlight.tsx` — timestamp 기반 위치 계산 + `ts.width()` 좌표계
- `PredictionGuide.tsx` — `ts.width()` 좌표계 수정
- `CandlestickChart.tsx` — 좌우 대칭 정렬

## 6. 위상군 자가 반성

### TimeHighlight 3단계 진화의 교훈
처음 `klineCount - 1` 기반으로 수정했으나 `priceStore`의 슬라이딩 윈도우 + `setData()` 패턴을 놓쳤음. 종일군의 실행 테스트 피드백이 없었다면 발견 불가능했을 것. **코드 분석만으로는 런타임 동작을 완전히 예측할 수 없다** — 이것이 종일군 테스트 단계(TCL#33)의 본질적 가치.

### shared 패키지 빌드 누락
`PAST_MULTIPLIER` 변경 후 `npm run build:shared`를 실행하지 않아 종일군이 변경 미반영 문제를 겪었음. 빌드군 하네스에 "shared 변경 시 `build:shared` 필수" 규칙 추가 완료.

### Protected Code 체계의 한계 인식
3단계 방어를 구축했지만, 빌드군이 파일을 전체 재작성하면 여전히 위험. 빌드 지시 시 위상군이 반드시 관련 보호 항목을 명시해야 함 — 체계가 자동으로 작동하는 것이 아니라 위상군의 책임.
