## 2. 완료된 작업 (상세)

### 2.1 Reflect vs Graphiti 분석

종일군이 DnT 프로젝트의 코드군이 TEMS에 Reflect(Hindsight)와 Graphiti를 적용했다고 보고. 위상군이 두 기술의 TEMS 대비 효용성을 분석:

- **Reflect (Hindsight)**: 현재 TEMS에서는 오버엔지니어링에 가까움
  - 이유: 규칙 수가 18개로 교차 패턴 발견의 통계적 기반 부족
  - TEMS의 위상적 추상화가 이미 1:1 변환을 수행하므로 추가 추상화 여지가 좁음
  - 규칙 100개+ 축적 + 다양한 도메인이 되면 그때 진짜 효용 발생
  - 현재는 RuleGraph의 키워드 겹침 기반 연결로 충분

- **Graphiti**: TEMS에 확실한 가치가 있다고 판단 → 구현 결정
  - Bi-temporal tracking (규칙 유효 기간)
  - Rule supersession (삭제 없는 무효화)
  - Point-in-time query (시간축 질의)

### 2.2 Evolution 4: TemporalGraph 구현

**파일:** `memory/tems_engine.py` (TemporalGraph 클래스 추가)

Graphiti(Zep, arXiv:2501.13956) 핵심 개념을 TEMS SQLite에 통합:

#### 새 테이블
- `rule_versions`: 규칙 수정 이력 (rule_id, version, snapshot_json, changed_fields, reason)

#### 새 컬럼 (마이그레이션)
- `memory_logs`: `valid_from`, `valid_until`, `superseded_by`
- `rule_edges`: `valid_from`, `valid_until`

#### 구현된 메서드
| 메서드 | 기능 |
|--------|------|
| `supersede_rule(old, new, reason)` | 규칙 대체 — 구 버전 보존 + 무효화 |
| `record_version(rule_id, changed_fields, reason)` | 수정 전 스냅샷 저장 |
| `get_rule_timeline(rule_id)` | 전체 시간축 이력 + 대체 체인 추적 |
| `query_at_time(timestamp, query, category)` | 특정 시점의 활성 규칙 검색 |
| `get_active_rules(category)` | 현재 유효한 규칙만 반환 |
| `detect_contradictions(rule, tags, threshold)` | 새 규칙의 기존 규칙 충돌 감지 |
| `invalidate_edge(a, b, type)` | 엣지 무효화 (삭제 아닌 valid_until) |
| `get_active_edges(rule_id)` | 유효한 엣지만 반환 |
| `get_temporal_stats()` | 시간축 통계 |

#### EnhancedPreflight 통합
- `TemporalGraph` 인스턴스를 내장
- preflight 검색 시 `valid_until`이 설정된(superseded) 규칙 자동 제외
- 캐스케이드 규칙에서도 superseded 규칙 제외

#### CLI 명령어 추가
```
temporal-stats       # 시간축 통계
timeline <rule_id>   # 규칙 시간축 이력
at-time <timestamp>  # 특정 시점 활성 규칙
supersede <old> <new> # 규칙 대체
```

#### 테스트 결과
- 기존 규칙 17개 전부 `valid_from = created_at`으로 백필 완료
- supersede 시나리오: 생성 → 대체 → 구 규칙 비활성화 → 신 규칙만 active 확인
- EnhancedPreflight temporal 필터링 정상 작동
- 기존 기능(preflight, cascade, prediction) 호환성 유지 확인

## 3. 핵심 기술적 결정

### 3.1 Graphiti를 Neo4j 없이 SQLite로 구현
- Graphiti 원본은 Neo4j 그래프 DB를 사용하지만, TEMS 규모(~18개 규칙)에서는 오버킬
- 기존 SQLite에 temporal 컬럼(valid_from/valid_until/superseded_by)과 rule_versions 테이블을 추가하는 경량 접근
- 확장성: 규칙이 1000개+ 되더라도 SQLite로 충분 (FTS5 BM25 검색은 수만 레코드까지 효율적)

### 3.2 Reflect는 현 단계에서 보류
- 규칙 수 부족 (18개) + 이미 위상적 추상화 → 추가 추상화 여지 좁음
- 규칙 100개+ 축적 시 재검토
