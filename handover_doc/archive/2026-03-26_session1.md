Plan Mode에서 탐색 에이전트 2개 + 설계 에이전트 1개를 사용하여 포괄적 설계 수행.

- `E:\MRV\src/` 내부에 npm workspaces 모노레포
- 3개 패키지: `packages/shared`, `server`, `client`

#### 모듈 10개 + MODULE.md 생성
- **shared:** types (market, prediction, round, score, user) + constants (game)

#### 게임 라운드 상태머신
```
IDLE(3s) → BETTING(30s) → LOCKED(5s) → LIVE(~145s) → RESOLVING(2s) → RESULT(10s) → IDLE
```

#### 다차원 예측 데이터 모델
- PredictionInput: band(low/high) + volatility(Level 2+) + timing(M2)
- ScoreBreakdown: bandScore(0-100) + volatilityScore(0-30) + narrowBandBonus(1.0-1.5x)
- UserProfile: level, xp, streak, 해금 체계

#### API 설계
- REST 6개: auth/init, round/current, round/predict, round/:id/result, user/profile, user/history
- WebSocket: price, kline, round:state, round:result

#### DB 스키마
- 4 테이블: users, rounds, predictions, round_results
- `src/server/src/db/migrations/001_init.sql`

#### 채점 알고리즘
- 밴드 적중: 60점 기본 + 좁은밴드 보너스 최대 40점
- 미적중: 거리 기반 지수 감소
- 변동성: 정확=30, 인접=10, 반대=0
- 좁은밴드 배율: 1.0-1.5x
- XP = score/2 × 연승보너스(최대 1.5x)

#### UI/UX 4개 화면 텍스트 와이어프레임
- Main Game, Active Round, Result, Profile

### 2.5 플레이테스트 버그 심층 분석

종일군 요청으로 빌드군 핸드오버 2개(13건 버그/개선)를 위상적으로 분석.

#### 5개 위상 패턴 발견

| 패턴 | 건수 | 핵심 |
|------|------|------|
| **상태 지속성 실패** | BUG-B, C | 유저 인지 상태는 게임 상태보다 느리게 전이 |
| **좌표계 불일치** | BUG-A, D, E, F | 같은 공간의 UI 요소는 같은 좌표계를 사용해야 함 |
| **유저 안내 부재** | BUG-G, ENH-D, E | 모든 상태에 SHOW/DO/BLOCK 명시 필요 |
| **입력 독점** | BUG-H | 오버레이 이벤트 분배 전략 필요 |
| **하드코딩** | BUG-01, 02, 03 | 다중 상태에서 단일 상태 가정 금지 |

#### 근본 발견
설계에 **유저 경험 계층(UX Layer)**이 없었음:
- 서버 상태머신은 있지만 유저 인지 상태 모델이 없음
- 데이터 모델은 있지만 상태별 CTA 매트릭스가 없음
- 모듈 구조는 있지만 시각 동기화 규약이 없음
- API는 있지만 인터랙션 분배 맵이 없음

- **"빌드군도 세션 종료 절차가 있나요?"** — MRV 인프라의 완성도를 확인하려는 디렉터로서의 점검.
- **"피드백을 위상군에게 해야 하나요?"** — 실행 레벨 버그는 빌드군 직접, 설계 레벨은 위상군이라는 라우팅 판단.
- **"코드를 수정한다고 해놓고 안 되어 있을 때가 있어요"** — AI 에이전트의 근본적 인지 한계(코드 작성 ≠ 작동 확인)를 정확히 감지. UX-VERIFY/FIX-VERIFY 프로토콜의 기원.
- **"에이전트들이 감정에 동조하는 경향"** — 동조 편향(Sycophancy) 문제를 직감적으로 인식. 독립 평가 에이전트 아이디어 제시. "너무 잔인한가?"라는 배려와 함께.

## 8. 위상군 자가 반성

### DESIGN에서 빠뜨린 UX Layer
이번 세션의 가장 큰 교훈: ADR, MODULE.md, 상태머신, API를 모두 설계했지만 **유저 경험 계층(UX Layer)을 설계하지 않았다.** 결과적으로 빌드군이 "기술적으로 작동하지만 유저가 사용할 수 없는" 코드를 생산.

이를 TCL #25로 영구 기록하여 향후 DESIGN에서는 UX Layer 4개 산출물(유저 인지 상태, CTA 매트릭스, 시각 동기화, 인터랙션 분배)을 반드시 포함.

### 빌드군 하네스의 도메인 공백
빌드군이 "DnT가 어떤 게임인지" 모른 채 코드를 작성한 것은 하네스 설계의 결함. 기술 프로토콜(BUILD, FIX, TEST)만 있고 도메인 컨텍스트가 없었음. 사후에 §6~§8을 추가하여 해결했지만, 처음부터 있었어야 함. TCL #26으로 향후 설계→빌드 전달 시 반드시 도메인을 함께 주입.

### 나선 시스템 실전 검증
Turn 1에서 IDEATE→ANALYZE→PLAN→DESIGN→BUILD까지 나선 5단계를 실전으로 돌렸다. Scale Pipeline(MODULE.md, Working Context, Scoped Delegation)이 빌드 단계에서 실제로 작동하는 것을 확인. 하지만 "설계의 완성도"에서 UX Layer 누락이라는 구멍을 발견 — 이것이 나선형 개발의 가치(반복마다 더 나아짐).
