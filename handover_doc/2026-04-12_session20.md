## 주요 결정 & Plan 편차 기록

### D1. python-frontmatter 하이브리드 델리게이션 (P2)
**상황**: deferred I2/I3/I4/I5 일괄 처리를 위해 python-frontmatter 전환 승인.
**결과**: library의 3개 sharp edge 발견:
  1. `frontmatter.loads`가 BOM을 처리하지 않음 (I4 수동 strip 필수)
  2. `Post.content`에서 trailing newline strip → body roundtrip 깨짐
  3. empty content 시 body separator 누락 → unclosed fence guard 재트리거

**해결**: sentinel 트릭 (`_BODY_SENTINEL = "\x00ATLAS_BODY_SENTINEL\x00"`) — library에 항상 non-empty body를 보내고 dumps 결과에서 실제 body로 치환. Body slicing은 수동 유지, YAML 파싱/덤프만 library 사용.

**검증**: 28/28 기존 테스트 전부 통과 (특히 Task 1.6 idempotency canary test).

### D2. Windows cp949 서브프로세스 인코딩 방어 (Task 4.1)
**상황**: hook template이 Korean 문자열을 print할 때 서브프로세스에서 UnicodeEncodeError 발생.
**근본 원인**: Python 3.12 Windows 기본 stdout = cp949, `subprocess.run(..., text=True)` = parent 로케일 사용.
**해결 (2면 패턴)**:
  - Child: `sys.stdout.reconfigure(encoding="utf-8")` at module load
  - Parent(test): `subprocess.run(..., encoding="utf-8")` (NOT `text=True`)
**적용 범위**: Task 4.1 hook + Task 4.3 hook + 플랜 파일 업데이트 완료.

### D3. Task 4.3 테스트 body sizing 플랜 버그 (Impl-3 편차)
**상황**: plan test의 L1/L2 body가 각 1 non-blank line → I3 ratio 1.0 (범위 [0.1, 0.5] 밖) → `check_all_invariants` warnings 비어있지 않음 → hook이 "atlas: clean" 대신 `<session-end-atlas-check>` 출력 → assertion 실패.
**근본 원인**: S19 Task 1.4와 동일 패턴 (plan test data와 실구현 invariant range 충돌).
**해결**: L1 body = 50 lines, L2 body = 10 lines (ratio 0.2 = I3 pass). 플랜 파일 사전 수정.
**교훈**: plan 작성 시 "healthy" fixture의 I3/I4/I5 invariant 통과 여부를 mental dry-run으로 검증해야 함 (S19 회고에서도 언급).

### D4. Task 3.2 import 전략 — 상대 import 성공
**상황**: `atlas_setup.py`가 처음으로 scripts/ 내 다른 모듈을 cross-import.
**결과**: `from .topology import ...` (상대 import) 1차 시도 성공. `scripts/__init__.py`가 Task 0.1에서 이미 생성되어 regular package로 작동.
**정정**: S19 handover에서 "implicit namespace package" 로 기재했으나, 실제로는 `__init__.py`가 있는 **regular package**. `pythonpath = [".."]`는 여전히 load-bearing (atlas/scripts가 atlas.scripts로 resolve).
